# Phase 6 Plan 02: Persistent SessionMemory, Memory Isolation, WriterPool, And RAG Validation

<objective>
把长文本运行所依赖的记忆、资源预算和检索输入从隐式内存状态升级为可恢复、可降级、可测试的运行时基础能力。
</objective>

<threat_model>
- Threat: Redis-backed memory stores stale or cross-task data. Mitigation: task-scoped keys, schema version, TTL/cleanup, and snapshot hash validation.
- Threat: cognee dependency failure blocks generation. Mitigation: adapter boundary with `MEMORY_ENABLED=false` and fallback backend behavior.
- Threat: WriterPool throttling causes deadlock. Mitigation: bounded acquire timeout, release in finally blocks, and test coverage for cancellation.
- Threat: unvalidated RAG query causes noisy retrieval or resource abuse. Mitigation: query length/quality caps and safe empty-result fallback.
</threat_model>
